# Настройка подчиненного ЦС на YEKT-RTR

## Настройка ЦС

Проверяем, чтобы был ```openssl```, если нет, ставим:

```
apt install openssl -y
```

Создаем структуру каталога для ЦС:

```
mkdir -p /sub/ca/{certs,crl,newcerts,private}
```

```
chmod 700 /sub/ca/private/
```

```
touch /sub/ca/index.txt
```

```
echo 1000 > /sub/ca/serial
```

```
echo 1000 > /sub/ca/crlnumber
```

Создаем конфиг ЦС:

```
nano /sub/ca/openssl.cnf
```

```
[ ca ]
default_ca = CA_default

[ CA_default ]
dir               = /sub/ca
certs             = $dir/certs
crl_dir           = $dir/crl
crlnumber         = $dir/crlnumber
database          = $dir/index.txt
new_certs_dir     = $dir/newcerts
serial            = $dir/serial

certificate       = $dir/certs/subca.crt
private_key       = $dir/private/subca.key

default_md        = sha256
default_days      = 825
copy_extensions   = copy
unique_subject    = no
policy            = policy_loose


[ policy_loose ]
commonName              = supplied

[ req ]
default_bits        = 4096
default_md          = sha256
distinguished_name  = req_distinguished_name

[ req_distinguished_name ]
commonName          = Common Name

[ v3_intermediate_ca ]
subjectKeyIdentifier   = hash
authorityKeyIdentifier = keyid:always,issuer
basicConstraints       = critical, CA:true, pathlen:0
keyUsage               = critical, digitalSignature, cRLSign, keyCertSign
```

Генерируем закрытый ключ ЦС:

```
openssl genrsa -out /sub/ca/private/subca.key 4096
```

Создаем сертификат ЦС:

```
openssl req -config /sub/ca/openssl.cnf -new -sha256 -key /sub/ca/private/subca.key -out subca.csr -subj "/C=RU/ST=Moskovskaya/L=Moskva/O=ColdEffort/CN=Yekaterinburg CA"
```

Переносим полученный ```subca.csr``` на ```MSK-DC1```:

```
scp subca.csr user@192.168.1.2:/home/user/subca.csr
```

На ```MSK-DC1``` одписываем основным ЦС:

```
openssl ca -config /root/ca/openssl.cnf -extensions v3_intermediate_ca -days 1825 -notest -md sha256 -in subca.csr -out subca.crt
```

Возвращаем подписанный сертификат ```subca.crt``` на ```YEKT-RTR```:

```
scp subca.crt user@192.168.2.1:/home/user/subca.crt
```

Кладем подписанный сертификат в ```/sub/ca/certs```:

```
mv subca.crt /sub/ca/cert/subca.crt
```

## Выпуск сертификатов для web-сервисов

### billing.effort.cool (сайт на YEKT-BILLING)

Генерим ключ:

```
openssl genrsa -out billing.effort.cool.key 4096
```

Создаем конфиг:

```
nano billing.effort.cool.cnf
```

```
[ req ]
distinguished_name = req_distinguished_name
req_extensions = server_cert

[ req_distinguished_name ]

[ server_cert ]
basicConstraints = CA:FALSE
keyUsage = digitalSignature, keyEncipherment
extendedKeyUsage = serverAuth
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = billing.effort.cool
IP.1 = 192.168.2.100
```

Делаем запрос сертификата:

```
openssl req -new -key billing.effort.cool.key -out billing.effort.cool.csr -config billing.effort.cool.cnf -subj "/C=RU/ST=Moskovskaya/L=Moskva/O=ColdEffort/CN=billing.effort.cool"
```

Подписываем сертификат:

```
openssl ca -config /sub/ca/openssl.cnf -extfile billing.effort.cool.cnf -extensions server_cert -notext -in billing.effort.cool.csr -out billing.effort.cool.crt
```

Собираем chain:

```
cat billing.effort.cool.crt /sub/ca/certs/subca.crt > billing.effort.cool.fullchain.crt
```

Затем переносим ```billing.effort.cool.key``` и ```billing.effort.cool.fullchain.crt``` на YEKT-BILLING и используем для настройки HTTPS.

# Работа с Wazuh

## Настройка HTTPS

После того, как выпустили сертификат, забираем ```wazuh.effort.cool.crt``` и ```wazuh.effort.cool.key```.

Далее переносим серт и ключ в нужную директорию:

```
cp wazuh.effort.cool.crt /etc/wazuh-dashboard/certs/
```

```
cp wazuh.effort.cool.key /etc/wazuh-dashboard/certs/
```

Вносим правки в конфиг:

```
nano /etc/wazuh-dashboard/opensearch_dashboards.yml
```

Меняем пути к файлу ключа и сертификата:

```
server.ssl.key: "/etc/wazuh-dashboard/certs/wazuh.effort.cool.key"
server.ssl.certificate: "/etc/wazuh-dashboard/certs/wazuh.effort.cool.crt"
```

Также меняем права на файл ключа и сертификата:

```
chown -R wazuh-dashboard:wazuh-dashboard /etc/wazuh-dashboard/
```

```
chmod -R 500 /etc/wazuh-dashboard/certs/
```

```
chmod 440 /etc/wazuh-dashboard/certs/wazuh.effort.cool.key /etc/wazuh-dashboard/certs/wazuh.effort.cool.crt
```

Перезапускаем Wazuh dashboard:

```
systemctl restart wazuh-dashboard
```

## Добавление правил

> Полный файл ```/var/ossec/etc/rules/local_rules.xml``` смотреть по [ссылке](local_rules.xml)

Открываем файл с локальными правилами:

```
nano /var/ossec/etc/rules/local_rules.xml
```

Правила для скана портов и брут SSH с Suricata:

```
<group name="routers">

  <!-- Сканирование портов в ЦОДе -->
  <rule id="86601" overwrite="yes" level="10">
    <field name="alert.signature">PORT SCAN SYN</field>
    <description>Router detected port scan</description>
  </rule>

  <!-- Брутфорс SSH в ЦОДе -->
  <rule id="300002" level="10">
    <field name="alert.signature">SSH BRUTE FORCE</field>
    <description>Router detected brute force attack</description>
  </rule>

</group>
```

Правило для буртфорса SSH:

```
<group name="custom,ssh">

  <!-- Базовое событие: неудачный вход -->
  <rule id="100002" level="5">
    <if_sid>5710</if_sid>
    <description>SSH failed login</description>
  </rule>

  <!-- Корреляция: брутфорс SSH -->
  <rule id="100003" level="10" frequency="5" timeframe="300">
    <if_matched_sid>100002</if_matched_sid>
    <description>SSH brute force detected (5 attempts in 5 minutes)</description>

  <mitre>
    <id>T1110</id>
  </mitre>
  </rule>

</group>
```

Правила для изменения системных файлов:

Сначала в основном файле конфигурации меняем частоту проверок системных файлов:

```
nano /var/ossec/etc/ossec.conf
```

Находим строки:

```
<syscheck>
<disabled>no</disabled>

<!-- Frequency that syscheck is executed default every 12 hours -->
<frequency>43200</frequency>
```

И изменяем строку ```<frequency>43200</frequency>``` на следующее значение:

```
<frequency>60</frequency>
```

Там же добавляем строки:

```
<directories check_all="yes" realtime="yes">/etc/shadow</directories>
<directories check_all="yes" realtime="yes">/etc/passwd</directories>
<directories check_all="yes" realtime="yes">/etc/ssh</directories>
```

Далее снова редактируем локальным правила:

```
nano /var/ossec/etc/rules/local_rules.xml
```

Добавляем строки:

```
<group name="fim">

  <!-- Изменения в /etc/passwd -->
  <rule id="400001" level="12">
    <if_group>syscheck</if_group>
    <field name="syscheck.path">/etc/passwd</field>
    <description>Critical file modified: /etc/passwd</description>
  </rule>

  <!-- Изменения в /etc/shadow -->
  <rule id="400002" level="12">
    <if_group>syscheck</if_group>
    <field name="syscheck.path">/etc/shadow</field>
    <description>Critical file modified: /etc/shadow</description>
  </rule>

  <!-- Изменения в SSH-конфигах -->
  <rule id="400003" level="12">
    <if_group>syscheck</if_group>
    <field name="syscheck.path">/etc/ssh</field>
    <description>Critical SSH config modified</description>
  </rule>

</group>
```

> Так как wazuh с ~~ебанцой~~ заковыркой, то отдельно прописываем в файле:

```
nano /var/ossec/etc/shared/default/agent.conf 
```

```
<agent_config>

<syscheck>
<disabled>no</disabled>
<frequency>60</frequency>
<scan_on_start>yes</scan_on_start>
<directories check_all="yes" realtime="yes">/etc/shadow</directories>
<directories check_all="yes" realtime="yes">/etc/passwd</directories>
<directories check_all="yes" realtime="yes">/etc/ssh</directories>
</syscheck>

</agent_config>
```

Перезапускаем Wazuh:

```
systemctl restart wazuh-manager
```

> После перезапуска нужно немного подождать (секунд 20-30) на сам перезапуск, сбор хэшей файлов и тд.

Потом идем на тачку с агентом, делаем

```
echo "hack" >> /etc/passwd
```

И смотрим на сервере

```
tail -f /var/ossec/logs/alerts/alerts.json
```

Можно принудительно запустить syscheck командой

```
/var/ossec/bin/wazuh-syscheckd -f
```

# Сайт с логами на YEKT-RTR

## Веб-сервер Nginx

Ставим пакеты:

```
apt install nginx -y
```

Создаем директорию для логов:

```
mkdir -p /var/www/logs
```

```
chown -R www-data:www-data /var/www/logs
```

```
chmod 777 /var/www/logs
```

Создаем конфиг для сайта:

```
nano /etc/nginx/sites-available/logs
```

```
server {
    listen 80;
    server_name _;

    root /var/www/logs;
    index index.html;

    location / {
        autoindex on;
    }

    allow 192.168.2.0/24;   # сеть Екатеринбурга
    deny all;

    types {
        text/plain log txt;
    }

    default_type text/plain;
}
```

Создаем ссылку на файл:

```
ln -s /etc/nginx/sites-available/logs /etc/nginx/sites-enabled/
```

Удаляем дефолтный сайт:

```
rm /etc/nginx/sites-enabled/default
```

Проверяем корректность конфигурации:

```
nginx -t
```

Перезапускаем nginx:

```
systemctl restart nginx
```

## Rsync для сбора логов

Устанавливаем пакет:

```
apt install rsync -y
```

Включаем rsync daemon:

```
nano /etc/rsyncd.conf
```

```
uid = www-data
gid = www-data
use chroot = no
max connections = 4
log file = /var/log/rsync.log

[logs]
    path = /var/www/logs
    comment = Logs storage
    read only = no
    auth users = loguser
    secrets file = /etc/rsyncd.secrets
```

Сохраняем пароль в файл:

```
echo 'loguser:RsynC_PA$$' > /etc/rsyncd.secrets
```

```
chmod 600 /etc/rsyncd.secrets
```

Запускаем:

```
systemctl enable rsync
```

```
systemctl restart rsync
```
